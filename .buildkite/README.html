Wave 0
触发入口

🔑 Initialize K8S key: initialize ▶

Pipeline 入口。读取环境变量，生成动态 pipeline YAML（pipeline.dynamic.yml），上传给 Buildkite 调度后续所有 step。

  • 1/1执行 pipeline.dynamic.yml，根据 SDK_PYTHON_VERSIONS 展开矩阵，上传 pipeline
（无产物，生成 pipeline 调度）
Wave 1
全平台并行构建
— 均依赖 Initialize，互相不等待
🚀 Build X86 Linux amd64 key: build-all-amd64 依赖 initialize ▶

使用 Bazel 编译 yuanrong 全量产物（C++/Python/Java/Go），打包主包 whl，上传 OBS。

  • [1/6]环境初始化，设置 PATH 使用 python3.11，配置 Bazel 远程缓存
  • [2/6]同步 git submodule（datasystem / frontend / functionsystem）
  • [3/6]Bazel 全量构建：make all，输出二进制和 wheel
  • [3.1]调用 scripts/package_yuanrong.sh 重新打包 whl（注入版本号）
  • [3.5]校验 Go plugin ABI 兼容性
  • [4/6]打印 Bazel 远程缓存命中率报告
  • [5/6]收集 artifacts，上传 Buildkite packages
  • [6/6]上传 whl 到华为云 OBS，写入 meta-data obs-urls.build-all-amd64
openyuanrong-*.tar.gz openyuanrong-*-cp311-*-linux_x86_64.whl
🐍 Build SDK X86 ×4 Linux amd64 key: build-sdk-amd64-{cp39|cp310|cp311|cp312} 依赖 initialize ▶

矩阵构建：对每个 Python 版本独立构建 openyuanrong_sdk wheel 并上传 OBS。与 Build X86 完全并行，互不依赖。

  • 1同步 submodule，调用 build_openyuanrong_sdk_wheels.sh
  • 2上传 Buildkite packages
  • 3上传 OBS，写入 meta-data obs-urls.build-sdk-amd64-{suffix}
openyuanrong_sdk-*-cp39-*-linux_x86_64.whl openyuanrong_sdk-*-cp311-*-linux_x86_64.whl （其他 cp 版本同理）
⚙️ SANDBOX_AMD64_SDK_STEP 控制哪个 suffix 进入 Build Image
🐧 Build arm Linux arm64 key: build-all-arm64 依赖 initialize ▶

与 Build X86 相同流程，在 arm64 agent 上运行，产出 arm64 主包 whl（cp39）。

  • [1–6]流程同 Build X86，PATH 使用 arm64 工具链（python3.9）
openyuanrong-*-cp39-*-manylinux_*_aarch64.whl
🐍 Build SDK arm ×4 Linux arm64 key: build-sdk-arm64-{cp39|cp310|cp311|cp312} 依赖 initialize ▶

在 arm64 agent 上构建 SDK wheel 矩阵，流程同 Build SDK X86。与 Build arm 完全并行。

openyuanrong_sdk-*-cp39-*-linux_aarch64.whl （其他 cp 版本同理）
⚙️ SANDBOX_ARM64_SDK_STEP（默认 cp39）控制哪个版本进入 Build Image arm
🍎 Build SDK macOS ×4 macOS arm64 key: build-sdk-macos-arm64-{cp39|cp310|cp311|cp312} 依赖 initialize ▶

在物理 macOS arm64 agent 上构建 SDK wheel（同一台 Mac mini 串行执行四个版本），产出 macOS 专用 wheel 并上传 OBS。

  • [2/4]ensure-macos-build-tools.sh 检查 Homebrew 工具链
  • [3/4]构建 SDK wheel（对应 Python 版本）
  • [4/4]上传 OBS，使用 OBS_PYTHON=python3 覆盖（macOS 无 /opt/buildtools 路径）
openyuanrong_sdk-*-cp311-*-macosx_*_arm64.whl （其他 cp 版本同理）
Wave 2
镜像构建 & 发布
— 每个 step 等待其对应的 Build + SDK 都完成
📦 Build Image K8S amd64 key: publish-sandbox-release-amd64 依赖 build-all-amd64 + build-sdk-amd64-cp311 ▶

下载主包 whl + SDK cp311 whl，用 Docker 构建三个 K8S 镜像并推送到 SWR（华为云镜像仓库）。SDK wheel 的 cp tag 决定镜像 Python 版本。

  • 1从 OBS 下载 openyuanrong-*.whl（主包 cp311，通过 meta-data obs-urls.build-all-amd64）
  • 2从 OBS 下载 SDK wheel（由 YR_K8S_IMAGE_SDK_WHEEL_PATTERN=openyuanrong_sdk*-cp311-*.whl 过滤）
  • 3build-images.sh：python_build_args_from_wheel() 从 wheel 名提取 cp311 → PYTHON_VERSION=3.11.9
  • 4构建 yr-base（Ubuntu + python 3.11.9）→ yr-controlplane（装主包+SDK）→ yr-node
  • 5构建 yr-runtime（仅 SDK），推送 SWR，输出 sandbox-release.json
yr-controlplane:<tag>-amd64 yr-node:<tag>-amd64 yr-runtime:<tag>-amd64
⚠️ SANDBOX_AMD64_SDK_STEP、SANDBOX_AMD64_SDK_SUFFIX、主包 PATH Python 版本 三者必须一致，否则 Build Image 失败
📦 Build Runtime amd64 ×4 K8S amd64 key: publish-runtime-amd64-{cp39|cp310|cp311|cp312} 依赖 build-all-amd64 + build-sdk-amd64-{cpXX} ▶

矩阵：对每个 Python 版本构建 yr-runtime 镜像变体（FaaS 运行时，支持多 Python 版本），推送 SWR。

yr-runtime:<tag>-amd64-cp39 yr-runtime:<tag>-amd64-cp310 yr-runtime:<tag>-amd64-cp311 yr-runtime:<tag>-amd64-cp312
📦 Build Image arm K8S arm64 key: publish-sandbox-release-arm64 依赖 build-all-arm64 + build-sdk-arm64-cp39 ▶

与 Build Image 相同流程，使用 arm64 主包（cp39）+ SDK cp39，构建 arm64 K8S 镜像（Python 3.9.18）。

yr-controlplane:<tag>-arm64 yr-node:<tag>-arm64 yr-runtime:<tag>-arm64
📦 Build Runtime arm64 ×4 K8S arm64 key: publish-runtime-arm64-{cp39|cp310|cp311|cp312} 依赖 build-all-arm64 + build-sdk-arm64-{cpXX} ▶

矩阵：对每个 Python 版本构建 arm64 yr-runtime 镜像变体，推送 SWR。

yr-runtime:<tag>-arm64-cp39 yr-runtime:<tag>-arm64-cp311 （其他 cp 版本同理）
🐍 Publish Wheels PyPI K8S key: publish-wheels-testpypi 依赖 所有 Build SDK（linux + macOS） ▶

将所有平台 SDK wheel 发布到 TestPyPI。受 ENABLE_PYPI_PUBLISH 开关控制，默认不启用。

所有 Linux amd64/arm64 + macOS SDK step 全部完成后才触发
Wave 3
Multi-arch 合并
— 等待所有 Build Image + Build Runtime 完成
📦 Build Manifest Multi-arch key: publish-sandbox-manifest 依赖 Build Image + Build Image arm + Runtime ×8 ▶

合并 amd64 + arm64 镜像，创建 Docker multi-arch manifest，推送到 SWR。同时打包 Helm chart。

  • 1docker manifest create 合并 yr-controlplane / yr-node / yr-runtime amd64 + arm64
  • 2打包 Helm chart（yr-k8s-*.tgz）
  • 3推送 manifest 到 SWR
yr-controlplane:<tag> (multi-arch manifest) yr-k8s-*.tgz (Helm chart)
Wave 4
集成测试
— 等待 Build Manifest 完成
🚀 Test K8S K8S Pod key: test-k8s 依赖 publish-sandbox-manifest ▶

在 K8S Pod 内运行 off-cluster smoke test：从 OBS 下载主包 + SDK whl，安装后用 pytest 验证 yr-k8s 基本功能。

  • 1下载主包 + SDK whl（由 SMOKE_SDK_WHEEL_PATTERN=openyuanrong_sdk*-cp311-*.whl 决定）
  • 2resolve_smoke_python() 从 wheel 名推导测试用 Python（cp311 → python3.11）
  • 3pip 安装 + pytest smoke 测试
⚙️ SMOKE_SDK_WHEEL_PATTERN 须与 SANDBOX_AMD64_SDK_STEP cp tag 保持一致

修改指导

🔄 升级主包 Python 版本（如 cp311 → cp312）

  1. 修改 pipeline.dynamic.yml:291：
    PATH=/opt/buildtools/python3.12/bin:...
  2. 修改 pipeline.dynamic.yml:61-62：
    SANDBOX_AMD64_SDK_STEP="build-sdk-amd64-cp312"
    SANDBOX_AMD64_SDK_SUFFIX="cp312"
  3. 修改 test_sandbox_k8s.sh:8,10：
    SDK_STEP_KEY=...cp312
    SMOKE_SDK_WHEEL_PATTERN=...cp312...
三处须同时修改，cp tag 必须一致，否则 Build Image 会因 OBS URL pattern 不匹配而失败

➕ 新增 SDK Python 版本（如加 3.13）

  1. 在 sdk_python_suffix()（L48）加入 python3.13) printf 'cp313' ;;
  2. 在 build-images.sh 的 python_build_args_from_wheel() 加入 cp313 映射
  3. SDK_PYTHON_VERSIONS 默认值会自动展开矩阵，无需额外改动
新增版本只影响 Build SDK 和 Build Runtime 矩阵，不影响主镜像版本

🐍 更换镜像内 Python 版本（amd64）

只需同步修改以下 3 个变量即可：

# pipeline.dynamic.yml
export PATH=/opt/buildtools/python3.XX/bin:...  # L291
SANDBOX_AMD64_SDK_STEP="build-sdk-amd64-cpXXX"  # L61
SANDBOX_AMD64_SDK_SUFFIX="cpXXX"                 # L62

# test_sandbox_k8s.sh
SDK_STEP_KEY="...cpXXX"           # L8
SMOKE_SDK_WHEEL_PATTERN="...cpXXX..." # L10
主包 PATH Python 版本 必须 = SDK cp tag，否则 yr-controlplane 镜像无法安装主包

📤 OBS 上传 Python 报错（ModuleNotFoundError: obs）

  1. Linux：OBS_PYTHON 默认指向 /opt/buildtools/python3.9/bin/python3（预装 esdk-obs-python）
  2. macOS step：已在调用处覆盖 OBS_PYTHON=python3
  3. 如需换其他路径：在对应 step 调用前加 OBS_PYTHON=/path/to/python3 bash .buildkite/upload_obs_artifacts.sh ...
esdk-obs-python 只装在 python3.9 环境里（历史遗留），切换主包 Python 版本不影响 OBS 上传

🍎 macOS SDK 构建失败

  1. 检查 scripts/ensure-macos-build-tools.sh 工具链是否完整
  2. 设置 SKIP_BREW_UPDATE=1 可跳过耗时的 brew update
  3. OBS 上传失败：确认 OBS_PYTHON=python3 已在 macOS step 显式设置（pipeline.dynamic.yml:1325）

⚡ 触发单独 Step 重跑

  1. 在 Buildkite UI 点击对应 Step → Retry
  2. 或通过 API：yr-bk trigger <build>
  3. Build Image 失败：先确认 Build X86 + Build SDK 对应版本均已通过且 OBS URL meta-data 已写入
